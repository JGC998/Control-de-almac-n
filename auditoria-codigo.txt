  if (error.code === 'P2025') return NextResponse.json({ message: '...' }, { status: 404 });
  if (error.code === 'P2002') return NextResponse.json({ message: '...' }, { status: 409 });

Archivos afectados (entre otros):
  - src/app/api/tacos/[id]/route.js
  - src/app/api/tarifas-rollo/[id]/route.js
  - src/app/api/productos/[id]/route.js
  - src/app/api/clientes/[id]/route.js
  - (y todos los demás [id]/route.js)

Solución: Añadir a src/lib/manejadores-api.js (ya existe el archivo):
  export function handlePrismaError(error) {
    if (error.code === 'P2025') return NextResponse.json({ message: 'Registro no encontrado' }, { status: 404 });
    if (error.code === 'P2002') return NextResponse.json({ message: 'Registro duplicado' }, { status: 409 });
    if (error.code === 'P2003') return NextResponse.json({ message: 'Registro tiene dependencias' }, { status: 409 });
    console.error(error);
    return NextResponse.json({ message: 'Error interno del servidor' }, { status: 500 });
  }

Uso en cada ruta:

    return handlePrismaError(error);
  }


--- 4. FORMATO DE RESPUESTAS API INCONSISTENTE ---

Problema: Distintas rutas devuelven formatos distintos para errores:
  - { message: 'Error' }      ← la mayoría
  - { error: 'Error' }        ← algunas
  - objeto directo             ← rutas GET
  Esto dificulta el manejo uniforme en el cliente.

Solución: Crear src/lib/api-response.js con un wrapper:
  export const ApiResponse = {
    ok:       (data, status = 200) => NextResponse.json(data, { status }),
    created:  (data)               => NextResponse.json(data, { status: 201 }),
    notFound: (msg = 'No encontrado') => NextResponse.json({ message: msg }, { status: 404 }),
    conflict: (msg = 'Conflicto')     => NextResponse.json({ message: msg }, { status: 409 }),
    badRequest:(msg)                  => NextResponse.json({ message: msg }, { status: 400 }),
    serverError:(msg = 'Error interno') => NextResponse.json({ message: msg }, { status: 500 }),
  };

--- 5. PATRÓN useMessage DUPLICADO EN 10+ COMPONENTES ---

Problema: Este bloque se repite en casi todos los componentes con formulario:
  const [message, setMessage] = useState(null);
  // ...
  setMessage({ type: 'success', text: 'Guardado correctamente' });
  // ...
  {message && <div className={`alert alert-${message.type}`}>{message.text}</div>}

Solución: Crear src/hooks/useMessage.js:
  export function useMessage(autoClearMs = 4000) {
    const [message, setMessage] = useState(null);
    const show = useCallback((type, text) => {
      setMessage({ type, text });
      if (autoClearMs) setTimeout(() => setMessage(null), autoClearMs);
    }, [autoClearMs]);
    return {
      message,
      showSuccess: (text) => show('success', text),
      showError:   (text) => show('error', text),
      showWarning: (text) => show('warning', text),
      Alert: () => message
        ? <div className={`alert alert-${message.type} mb-4`}>{message.text}</div>
        : null,
    };
  }

Uso:
  const { showSuccess, showError, Alert } = useMessage();
  // ...
  <Alert />

--- 7. INCONSISTENCIA EN PARAMS (await params) ---

Problema: Los [id]/route.js del proyecto mezclan formas de leer params:
  - const { id } = await params;                     ← correcto (Next.js 15+)
  - const { id } = await paramsPromise;              ← variante válida pero inconsistente
  - const { id } = params; (sin await)               ← incorrecto en Next.js 15+

Solución: Estandarizar a siempre usar:
  const { id } = await params;


--- 8. revalidatePath INCONSISTENTE ---

Problema: Unas rutas llaman revalidatePath tras mutaciones, otras no.
  - src/app/api/materiales/route.js → sí lo usa
  - src/app/api/productos/route.js → sí lo usa
  - src/app/api/tacos/route.js → NO lo usa
  - src/app/api/tarifas-rollo/route.js → NO lo usa

Impacto: Con SWR en cliente esto no es crítico (el cliente revalida via mutate()), pero
es inconsistente y puede afectar si se añaden server components en el futuro.

Solución: O añadirlo sistemáticamente a todas las rutas mutantes, o eliminarlo de todas
(dejando solo la revalidación por SWR en cliente). Elegir una estrategia y ser consistente.


--- 9. SERIALIZACIÓN DE DECIMALES DISPERSA ---

Problema: Cada ruta que devuelve precios/pesos convierte Decimal de Prisma a Number manualmente:
  precioUnitario: p.precioUnitario ? Number(p.precioUnitario) : 0,
  pesoUnitario:   p.pesoUnitario   ? Number(p.pesoUnitario)   : 0,
  Esto aparece repetido en productos/route.js, materiales/route.js, etc.

Solución: Añadir a src/lib/manejadores-api.js:
  export function serializeDecimals(obj, fields) {
    return { ...obj, ...Object.fromEntries(fields.map(f => [f, obj[f] ? Number(obj[f]) : 0])) };
  }
  // Uso: serializeDecimals(producto, ['precioUnitario', 'pesoUnitario'])
